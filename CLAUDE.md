## プロジェクト概要

ネオン・ベクター表現のホードサバイバル（Vampire Survivors系）。自機を動かすと武器が自動発射し、「侵食データ」の群れを掃討、強化を選んで強くなり、**5分間生存**で勝利。グラフィックは全てコードで描く発光図形（スプライト画像なし）。

1.44MB GAME_DEV CONTEST 2026（2P Game Arcade）応募作。**締切：2026-09-04 23:39**。

## 絶対制約（ABSOLUTE — これらは何より優先）

- **容量**：解凍後の配布物の総容量を **1,474,560 バイト以下**に保つ。
- **形式**：独立して動くネイティブ実行ファイル。**ブラウザゲーム禁止**。
- **対応OS**：開発は macOS、提出に向けて **macOS と Windows x64 の両方で動く**こと。
- **オリジナル**：2026-05-23 以降に新規制作したコードのみ。

## 技術スタック

- 言語/ライブラリ：**C + raylib**（必要なら sokol への乗り換えも検討可）。
- 描画：raylib の加算合成（`BLEND_ADDITIVE`）でネオン表現。ブルームは描画先テクスチャ＋フラグメントシェーダ1枚（任意）。
- 音：**トラッカー音源（.xm 等）かソフトシンセ生成のみ**。
- フォント：小型ビットマップフォント。
- ビルド：静的リンク ＋ `strip` ＋ `-Os`/LTO ＋ 未使用raylibモジュール無効化。
- macOS は arm64 / x86_64 を**別ビルド**にする（ユニバーサルバイナリは容量を二重に食うので避ける）。

## コード方針

- C99、依存最小、1ファイル肥大化を避けてモジュール分割（player/enemy/weapon/spawn/render/audio など）。
- コメントは簡潔に。マジックナンバーは調整しやすいよう定数にまとめる（チューニング表参照）。
- 固定タイムステップ寄りの更新ループ。dt はクランプ（上限 ~0.04s）。
- 当たり判定は素朴な距離判定で開始。重くなったら簡易グリッドで空間分割。

## ビジュアル区別ルール（ALWAYS）

「ぱっと見で敵か味方か判別できる」ことを最優先する。新しい要素を追加・変更するときは必ず以下を守る：

**役割別カラーパレット（厳守）**
- 🟦 **クール系（シアン / 青 / バイオレット / 白）** = プレイヤー本体・自分の武器・自分の弾
- 🟢 **グリーン系** = アイテム・経験値ジェム・回復系強化・回収範囲強化
- 🟡 **イエロー / オレンジ系の高彩度** = 自分の弾（Pulse Bolt）・自軍の輝き
- 🔴 **ウォーム系（赤・オレンジ・ピンク・マゼンタ）** = **敵すべて**・敵弾
- 🟨 **金 / 暖色の純黄** = ボス専用（FORMAT）

**役割別シェイプ**
- ジェム = **菱形**（回転 or 脈動）。円は使わない（敵と混同するため）
- プレイヤー弾 = **流線型・光条**（DrawLineEx）。円は弾以外に使う
- 敵 = 円・四角・三角・十字など多様。プレイヤー武器と同じ形を避ける

**やってはいけないこと（NEVER）**
- 緑色の敵を作らない（ジェムと混同する）
- シアン/青色の敵を作らない（プレイヤーと混同する）
- 黄色の敵を作らない（Pulse弾と混同する）→ ボス専用
- 赤系の自軍要素を作らない（敵と混同する）

**新しい要素を追加するときの判定フロー**
1. それは敵性 or 友好的？ → ウォーム系 or クール系を選ぶ
2. 既存の同色要素と混同しないか？ → 形（円/菱/光条等）で差別化
3. 加算合成（BLEND_ADDITIVE）で重なったとき、識別できるか？

要素を追加・色変更する際は、必ずこのルールに照らして自己レビューすること。

## ビルド/実行（例：要調整）

```sh
# macOS（raylib を brew や自前ビルドで用意）
cc -std=c99 -Os -flto src/*.c -o disk_survivor \
   $(pkg-config --cflags --libs raylib) && strip disk_survivor
ls -l disk_survivor   # ← サイズ確認を習慣化

# Windows は MinGW / MSVC で raylib を静的リンク
```
