# DISK SURVIVOR（仮題）— ゲーム設計ドキュメント v1

> 1.44MB GAME_DEV CONTEST 2026（2P Game Arcade）応募作の設計書。
> 締切：2026年9月4日 23:39 ／ 提出物：独立実行ファイル（Mac/Windows）

---

## 1. 概要

ネオン・ベクター表現のホードサバイバル（Vampire Survivors系）。プレイヤーは自機を動かすだけで、武器は自動で発射。湧き続ける「侵食データ」の群れを掃討し、強化を選んで雪だるま式に強くなり、**5分間を生き延びる**。すべてコードで描く発光図形（スプライト画像ゼロ）で、フロッピー1枚＝1,474,560バイトに余裕で収める。

### コンテスト要件との適合

| 要件 | 対応 |
|---|---|
| 容量 1.44MB 以下（解凍後） | 素材ゼロ＋合成音で余裕で収まる |
| 独立実行ファイル / ブラウザ禁止 | raylib等でネイティブ実行ファイルをビルド |
| 公開日(5/23)以降に新規制作 | 今から制作開始 |
| エンジン自由 | raylib + C（ランタイム込みでも制限内） |
| 審査基準：①完成 ②容量内 ③面白い | スコープを絞り完成最優先／鉄板の面白さ／極小容量 |

---

## 2. コアループ

移動（自機）→ 自動攻撃で撃破 → 経験値ジェムを回収 → レベルアップで強化を選択 → さらに強く → 敵が激化 → 5分耐えて勝利。

操作：移動のみ（PC＝WASD/矢印、タッチ＝押した方向へ移動）。攻撃・狙いは全自動。

---

## 3. アートディレクション

- **ネオン・ベクター**：発光する図形・線・粒子だけで描画。スプライト画像は使わない。
- 加算合成（additive blending）で、図形が重なると自然に発光する。
- 仕上げにブルーム（発光）シェーダを1枚かけて全体を底上げ。
- 限定パレット（暗い背景＋シアン/マゼンタ/アンバー/グリーンの少数色）で統一感を出す。
- 「画」の良さは高精細ではなく**密度と手触り（ジュース）**で作る：撃破バースト、画面シェイク、ヒットフラッシュ、レベルアップ演出、ジェム回収の吸い込み。

---

## 4. 1プレイの構造（5分生存）

| 時間 | 内容 |
|---|---|
| 0:00–1:30 | 導入。基本の追跡敵。最初の数回のレベルアップ |
| 1:30–3:00 | 敵の種類と密度が増える |
| 3:00–4:00 | エリート／分裂敵が登場、強度上昇 |
| 4:00–5:00 | **クライマックス**：総攻撃＋ボス「FORMAT」。耐え切れば勝利 |

- 勝利条件：5:00 まで生存（＝「ディスク復旧成功」）。
- 敗北条件：HP が 0。リザルトに生存時間・撃破数・到達レベルを表示。
- レベルアップは1プレイで **12〜15回**来るテンポに調整（5分で強くなりきる）。

---

## 5. テーマ（軽い味付け）

「ディスク／コンピュータの内部で、侵食してくるデータと戦う」。ストーリーは作らず、見た目と名前の枠だけ。

- 敵＝破損データ、ウイルス、バッドセクタ。
- クライマックス＝ディスク全面崩壊（`FORMAT`）。
- これによりコンテストのフロッピー文脈とも噛み、敵に個性が出る。

---

## 6. コンテンツ範囲（v1）

### 武器・能力（4〜5種、各3段階強化）

1. **Pulse Bolt**（初期）：最寄りの敵へ自動発射。強化＝連射/多重/貫通
2. **Orbiters**：自機の周囲を回る発光オーブ（接触ダメージ）
3. **Beam**：周期的に回転するレーザー
4. **Nova**：周期的に自機周囲へ放つAoE（近接の群れを一掃）
5.（予備）**Spark Mines**：設置型の追尾火花

### パッシブ強化

移動速度／回収範囲／最大HP／ダメージ／連射速度／経験値取得量

### 敵（4〜6種＋ボス）

1. **Bit**：基本の追跡（弱・数で攻める）
2. **Fragment**：高速・脆い
3. **Packet**：低速・硬い
4. **Glitch**：不規則移動
5. **Splitter**：撃破時に小型に分裂
6.（予備）**Bad Sector**：固定の湧き口／障害
- **ボス FORMAT**：ラスト1分に出現。大型・攻撃パターンあり

---

## 7. メタ進行

**v1は最小**：ベストタイム／スコア表示のみ。リプレイ性は「強化ビルドの組み合わせ」と「難易度」で確保。
時間が余れば、武器やキャラのアンロックを後付け（スコープ外＝最初に削る候補）。

---

## 8. 技術スタック

- **エンジン**：raylib + C（Mac/Windowsネイティブ）。最大限の余裕や尖りが欲しければ sokol、C以外なら raylib の Zig/Odin/Rust バインディングも可。
- **描画**：raylib の加算合成（`BLEND_ADDITIVE`）。ブルームは描画先テクスチャ＋フラグメントシェーダ1枚。
- **音**：トラッカー音源（.xm 等）またはソフトシンセ生成。録音音源（mp3/ogg）は容量を食うので不使用。
- **フォント**：小型ビットマップフォント。
- **ビルド対象**：macOS（arm64 と x86_64 は別ビルドにして容量節約）、Windows x64。
- **最適化**：静的リンク＋strip＋`-Os`/LTO、未使用raylibモジュールの無効化。

## 9. 容量予算（目安）

| 要素 | サイズ |
|---|---|
| ランタイム/バイナリ（最大の変動要素） | 約 1.0–1.4 MB |
| シェーダ | 数 KB |
| 音（トラッカー） | 約 30–100 KB |
| フォント | 約 5–20 KB |
| コード | 誤差 |
| **合計** | **1,474,560 バイトに収まる** |

※ 敵や弾が何百体いても容量には影響しない（メモリ/CPUの話）。容量の敵は録音音源のみ。

**削る候補（時間が足りなければ最初に切る）**

## 12. リスクと対策

- **raylibバイナリが1.44MBに近い** → strip/LTO/未使用モジュール無効化。ダメなら sokol へ。
- **大量の敵で処理落ち** → 敵数の上限＋簡易な空間分割（グリッド）。
- **スコープ肥大** → v1を固定。上記「削る候補」を即カット。
- **Windowsビルドが最後に壊れる** → 早期（1〜2週）に両OSで動かし、以後も定期確認。
- **音で容量超過** → トラッカー/合成のみ。録音音源は使わない。
